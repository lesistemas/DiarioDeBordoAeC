Notas de Processo
①
Início do Robô Ponto de entrada. Executado continuamente em loop.
②
Carregar Parâmetros Carrega do AutomationRoom: credenciais, filas, lista não passíveis, parâmetros de RabbitMQ e URLs dos sistemas.
③
Buscar Caso — CM O robô verifica as filas do Case Manager por ordem de prioridade. Se não há caso disponível, aguarda e tenta novamente.
④
Coletar Cartão e Transações Registra IdCaso, fila selecionada, últimas 4 do cartão e transações vinculadas ao alerta.
⑤
Regra de Suspeita Aplica a regra de transações suspeitas antes de consultar o PW. Se houver erro ou flag de humano, encaminha diretamente.
⑥
Consultar PW / Vision Consulta o sistema PW/Vision para enriquecer o caso com dados completos do cliente e cartão.
⑦
Mesa de Decisão Busca os telefones do cliente na Mesa de Decisão. Obrigatório quando cadastro ou alteração é recente (<2 meses).
⑧
Lista Negativa — Redis Consulta o Redis para verificar se o telefone está na lista negativa. O Redis é alimentado por outro robô via Mesa de Decisão.
⑨
Motor de Regras O motor de regras decide o destino do caso após avaliar todas as condições em ordem fixa.
⑩
Registrar Caso (Descarte) Registra o caso no Gateway API com motivo de descarte.
⑪
Salvar Descarte — CM Salva a resposta de descarte no Case Manager, finalizando o caso no sistema.
⑫
Registrar Caso (Humano) Registra o caso no Gateway API com motivo de tratativa humana.
⑬
Disparo Tratativa Humana Registra o disparo do tipo Tratativa Humana via Gateway API.
⑭
Publicar RabbitMQ (Humano) Publica o evento de tratativa humana no RabbitMQ para sistemas downstream.
⑮
Preventivo P Aplica o bloqueio preventivo P no PW quando a configuração da fila exige. Garante que o cartão seja bloqueado antes do mailing.
⑯
Salvar Cartão — CM Salva a resposta do cartão no Case Manager com a categorização correta para o caso automático.
⑰
Registrar Caso (GW) Registra o caso completo no Gateway API com transações, resposta e motivo.
⑱
Disparo Mailing Registra o disparo de mailing no Gateway API.
⑲
Publicar RabbitMQ (Auto) Publica o payload operacional completo no RabbitMQ para sistemas downstream.
⑳
Fim do Caso O caso é encerrado. O robô retorna ao loop para buscar o próximo caso disponível.

02 · Motor de Regras Detalhado
Dados / Entrada
Verificação
Descarte

Notas de Processo
R0
Dados Coletados BlockCode, telefone, transações, datas cadastrais — prontos para avaliação das regras.
R1
Não Passível / Bloqueado / Bypass Compara o BlockCode com a lista de não passíveis. Também verifica se o cartão está bloqueado ou se o bypass de 2 dias está ativo.
R2
Fila Exige Humano? Verifica se a configuração da fila tem o flag ExigeHumano ativo. Se sim, o caso vai diretamente para analista.
R3
Regra de Suspeitos Exige Humano? Avalia se a regra de suspeitos retornou HouveErro=true ou SegueTratativaHumano=true.
R4
Cliente tem Telefone Válido? Verifica se o PW retornou um telefone válido. Sem telefone, o comportamento depende da configuração RegraSemTelefone.
R4b
Sem Telefone — Fila Teto? Verifica se a configuração da fila exige que o cartão seja salvo no CM por não ter telefone válido. Controla o comportamento de encerramento.
R5
Mesa de Decisão — Tel. Válido? Mesa de Decisão: verifica por CPF se o cadastro ou a alteração tem menos de 2 meses. Se o telefone não for validado, vai para humano.
R6
Telefone na Lista Negativa? Verifica no Redis (lista negativa) se o telefone está bloqueado. O Redis é alimentado por outro robô da esteira.
R7
Preventivo P (PW/Vision) Aplica o bloqueio preventivo P no PW quando a configuração BloqueioPreventivoP=NoInicio está ativa para a fila.
R8
Disparo de Mailing (GW) Registra o disparo de mailing no Gateway API. Indica que o caso foi resolvido automaticamente.
R9
Publicar RabbitMQ Publica o evento no RabbitMQ. Tanto casos automáticos quanto humanos publicam aqui — é a saída operacional final.
DESC
Descarte — Coluna Esquerda Caso descartado: block code não passível, cartão bloqueado ou bypass ativo. Salva descarte no Case Manager e segue para próximo caso.
SEM TEL
Sem Telefone Sem telefone válido e fila não exige Salvar Cartão: salva a resposta "Sem Telefone" no Case Manager e segue para o próximo caso, sem tratar como humano nem automático.
HUM
Tratativa Humana — Coluna Direita Qualquer regra pode resultar em tratativa humana. O caso é encaminhado para fila de analistas, disparo registrado e evento publicado no RabbitMQ.

03 · Regras de Transações Suspeitas
Entrada / Saída
Verificação
Mensageria automática
Tratativa humana
Subcritério exceção
Notas de Processo
E
Transações Recebidas A regra recebe a lista completa de transações do cartão coletadas no Case Manager. Decide quais incluir no alerta e se o caso vai para mensageria automática ou para um analista.
F1
Filtro SUSPEITA O sistema lê o campo "Análise" de cada transação e mantém apenas as que contêm a palavra SUSPEITA. Transações com outros status são ignoradas.
D1
Alguma Transação Suspeita? Se nenhuma transação foi classificada como suspeita, não há base para acionar o alerta. O caso é encaminhado a um analista para avaliação manual.
H1
Humano — Nenhuma Suspeita Sem transações suspeitas identificadas — o caso não tem base para mensageria. Encaminhado ao analista.
D2
Suspeita é do Dia de Hoje? O sistema verifica se alguma das transações suspeitas ocorreu no dia de hoje. Isso separa casos recentes de casos de dias anteriores, que têm tratamento diferente.
A1
Suspeitas Antigas Todas as suspeitas são de datas anteriores — caso antigo ainda em aberto. A regra seleciona apenas a transação suspeita mais recente para compor o alerta, evitando sobrecarregar o cliente com histórico.
D3
Quantas Suspeitas Hoje? O limite de 3 transações garante que a mensagem ao cliente seja objetiva e legível. Acima de 3, há risco de fraude coordenada — o analista precisa avaliar.
A2
Casos do Dia (1–3) Até 3 transações suspeitas no dia. Volume adequado para o cliente avaliar. Todas são incluídas na mensagem de alerta.
EX
Avalia Exceção (>3 suspeitas) 3 subcritérios: mesmo EC, valores idênticos, horários consecutivos <5 min. Se todos atendidos = duplicidade técnica → mensageria. Se falhar = humano.
C1
Mesmo Estabelecimento? Verifica se todas as transações suspeitas foram feitas no mesmo estabelecimento. Usa o Código do Estabelecimento como identificador primário; nome como fallback.
C2
Valores Idênticos? Verifica se todos os valores das transações suspeitas são idênticos. Valores iguais repetidos no mesmo estabelecimento são padrão comum de duplicata técnica.
C3
Horários Consecutivos <5 min? Verifica se o intervalo entre cada compra consecutiva é inferior a 5 minutos. Esse padrão indica duplicidade técnica (retentativa automática do sistema) e não fraude real.
D4
3 Critérios Atendidos? Os 3 critérios precisam ser verdadeiros ao mesmo tempo. Se todos forem atendidos, é provável duplicidade técnica — segue mensageria com as 3 últimas transações. Se qualquer critério falhar, o volume acima de 3 indica risco real e o caso vai para análise humana.
A3
Exceção Válida — Todas as Suspeitas Padrão de duplicidade técnica confirmado. O sistema segue com mensageria, incluindo as 3 transações mais recentes no alerta ao cliente.
H2
Humano — Risco Alto Com mais de 3 suspeitas no dia e sem padrão de duplicidade, o risco é alto o suficiente para exigir avaliação humana. O caso é encaminhado ao analista com todas as transações identificadas.
S
Segue Mensageria Automática O resultado da regra é uma lista de transações selecionadas (máx. 3) que serão incluídas no alerta enviado ao cliente. O robô segue para montar e disparar a mensagem.
